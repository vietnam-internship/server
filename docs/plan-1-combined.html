🔒 lock held for this whole span — its length depends on PG latency (seconds, not milliseconds)
Every other request R2, R3, … for the same slot waits in the lock queue for this entire span. The moment the PG slows down, bookings for this slot stall entirely; worst case = waiters × 30 s timeout.
The failure mode is worse too — payment succeeded + response lost = "money out, no reservation" → recovery is a refund + CS ticket.
